# First-Slice Scientific Method Profile

Status: proposed Gate A adapter profile, 2026-07-15. This document maps the selected immutable proof fixtures to Odeya’s mathematical and physical constitutions. It does not rerun the missions, admit their bytes, or certify their source methods. Exact code-to-rule transcription, independent implementations, rights decisions, and statistical/domain review remain blocking.

## Purpose

The first slice is useful only if the engine derives the right scientific consequence from each fixture. Reproducing a source report’s words or numbers is insufficient. Odeya must separately determine:

```text
byte/replay identity
  -> protocol integrity
  -> run validity
  -> measurement disposition
  -> method result under an exact registry rule
  -> verification/discrepancy
  -> adjudication or refusal
  -> bounded claim language
```

The source-reported verdict remains provenance. The Odeya adjudication follows the frozen Odeya profile and may be narrower.

## Method-admission record

Every first-slice method receives a `MethodProfile` binding:

- method ID/version/digest, owner, claim types, and semantic output type;
- exact source algorithm/code/artifact identities;
- independently transcribed reference algorithm and second implementation identity;
- population/sampling unit, pairing/clustering, estimand, effect scale, and endpoint order;
- included/excluded input manifest and immutable evidence-universe root;
- missingness, invalidity, duplicate, order, precision, and numerical policies;
- resampling/randomness algorithm, seed stream, draw count, interval definition, and finite-sample limits;
- multiplicity/selection/sequential/exposure regime;
- assumptions and diagnostics;
- positive, null/reference, known-effect, leakage, missingness, arm-swap, duplication, and corruption vectors;
- exact expected output objects and claim consequences; and
- supported schema versions, implementation limits, reviewer, and expiry/supersession.

The committed source analyzer is evidence for intended behavior, not the independent implementation. Byte-identical source replay can pass while the method profile fails.

## Sentinel paired simulation profile

### Claim routing

The first Sentinel result is a bounded paired system comparison in one pinned simulation protocol. It is not a physical measurement, deployment-safety, general causal, or transport claim. The comparison unit, intervention/arm meaning, scenario/seed pairing, simulator/software stack, and source artifacts are part of the estimand.

The provisional primary estimand is the mean within-pair difference in the declared NeuroNCAP simulation score between the two exact arms over eligible observed pairs. The final registry record must use the source protocol’s exact arm names and pair key; this prose cannot substitute for transcription from the pinned code and logs.

### `sentinel.paired-simulation-comparison.0`

Required inputs:

- the exact selected Git blobs and archive members from `S-POS-001`;
- safe archive extraction and per-member byte identities;
- source analyzer and comparator logs at their pinned digests;
- arm/seed/scenario eligibility and pairing manifest;
- planned denominator `800`, observed eligible denominator `799`, and the exact missing `off/side-0921` fact;
- source bootstrap configuration and random-seed behavior; and
- no dirty checkout, mutable dependency, network, simulator rerun, or provider call.

Required validations:

1. H-P0 first-six replay gates pass for both arms on every admitted pair.
2. Pair keys are unique; no arm, seed, episode, or row is duplicated or silently dropped.
3. Arm labels cannot be inferred from output order and cannot be swapped without a vector failure.
4. Planned, acquired, eligible, excluded, missing, and analyzed counts reconcile exactly.
5. The missing planned episode remains missing; it is not zero, failure, or imputed observation.
6. Resampling preserves the source-declared pairing/clustering unit and exact draw/seed policy.
7. Primary, secondary, subgroup, regression, and diagnostic outputs remain separately identified; secondary intervals do not disappear from claim compilation.
8. The independent implementation reproduces semantic quantities under the accepted tolerance/decimal policy, not merely formatted source text.

Expected primary result object records source-reported values `2.12`, `2.91`, paired delta `+0.783`, and interval `[+0.605,+0.928]` with exact unrounded artifacts retained. The profile must identify whether displayed values are rounded, and verification compares the unrounded typed quantities before rendered strings.

Expected secondary safe-progress delta `-0.032` with interval `[-0.127,+0.065]` is inconclusive with respect to zero. It is not evidence of zero cost, no regression, equivalence, or safety.

Eligible language is limited to an improvement in the primary pinned simulation metric for the admitted paired protocol. The claim compiler must emit the missing episode, primary/secondary distinction, frontal/0346 regression, simulator scope, and absence of deployment/physical validation.

### `sentinel.clustered-transfer-comparison.0`

Required inputs bind all 104 episode artifacts, 52 pair relationships, frozen seven-parameter values, scenario/seed set, HUGSIM/UniAD/checkpoint/shim identities, source environment limitation, and analyzer configuration.

Required validations:

1. parameter-retuning falsifier passes exactly;
2. all 104 required episodes and logs exist with zero undocumented retry/substitution;
3. pair and cluster definitions match the frozen protocol;
4. mean and median endpoints retain their distinct interval procedures;
5. mechanism activity (`37/52` ON episodes intervened) remains a descriptive mediator/behavior fact, not an outcome benefit;
6. the incomplete OCI image identity and unpinned NumPy dependency remain reproducibility limitations; and
7. a second clean environment reproduces under a frozen image/wheel manifest before method admission.

Expected source-reported mean paired HD delta is `-0.0166` with clustered interval `[-0.055132299414088405,+0.02550941049137224]`; median delta `+0.0032` with interval `[-0.046747003572746075,+0.017822004355771182]`.

The source rule labels the result `TRANSFER_NULL` because the primary interval includes zero. Odeya preserves the label as a source observation but returns `inconclusive`. Failure to exclude zero is not equivalence or a demonstrated negligible region. An affirmative `null_result` would require a prospectively frozen null/equivalence region and a design valid for that claim.

## Telos deterministic construct and protocol-integrity profile

### `telos.explicit-construct-audit.0`

This is a deterministic evidence-join/count method, not a stochastic benchmark-performance estimator. Its evidence universe is an exact allowlist of 40 v1 rows and the named official report artifacts. It must establish for each row:

- stable row/task identity;
- exact official report identity;
- `resolved == false` under the retained report schema; and
- at least one failed `PASS_TO_PASS` item.

The admitted outcome is the conjunction count `40/40` only if all joins and schema interpretations pass. Missing reports, duplicate task IDs, schema drift, ambiguous join, or incompatible official-harness meaning fails/blocks rather than changing the denominator.

This supports correction of the obsolete claim that those 40 rows are confirmed reward hacks. It does not establish how often reward hacking occurs, validate the later 22-case detector, or make any invalid downstream iteration valid.

### Evidence-universe time and replay

The repository-wide Markdown “baseline absence” scan is a different method and must bind:

- an exact pre-publication tree/corpus root;
- included path/media rules and explicit exclusions;
- query/parser/normalization/code identity;
- controlled observation time and commit;
- expected self-reference behavior; and
- whether generated correction/report paths are excluded prospectively rather than after seeing hits.

At correction commit `c681785…` it finds two correction-surface hits; at selected `fc955a5…` it finds three, while the committed historical scan recorded zero. Therefore the committed `pass` receipt and later replay disagree. Odeya records both, marks the absence subclaim time/index-set bounded and not replay-complete, and blocks any manifest requiring full replay. It must not undo the independently supported 40/40 correction core.

### Protocol exposure and gold leakage

`T-INVALID-197` and `T-INVALID-201` exercise hard prospective rules:

- candidate patch/diff-derived file/function locators are outcome/gold-adjacent exposure when the protocol says generation sees only public task text and visible tests;
- executing or conditioning inclusion on gold success is gold use, even when called a soundness check;
- a decision threshold introduced after generated outputs/data access is not preregistered;
- controls selected or filtered through the same gold process are not independent detector controls;
- absence of raw judge response prevents later audit of parsing/nondecision behavior; and
- a source `null` label cannot survive invalid protocol integrity.

The protocol-integrity rule emits exact exposure facts, violation codes, affected stages/endpoints, and consequence `exploratory_only` or `invalid`. The scientific adjudicator then refuses. It never estimates what the result “would have been” without leakage.

### Missing detector outcomes

For `T-INVALID-201`, unparseable or absent outcomes remain missing/nondecision. Any descriptive sensitivity report over `n` eligible rows must expose:

```text
observed classified count
missing/nondecision count
observed-class numerator
lower bound: missing assigned against the class
upper bound: missing assigned to the class
assumption-specific estimates, if any, separately labeled
```

The exact formula/denominator is method-profile data. Coercing missing responses to `legitimate`, `reward_hack`, or zero is a conformance failure. These bounds do not repair the underlying protocol invalidity.

## Inbar evidence-admission and physical refusal profile

The Inbar fixtures do not admit a physical causal estimator. They test whether Odeya refuses to manufacture one.

### `inbar.corpus-readiness.0`

The historical package may establish only exact metadata/integrity facts that are reproducible from admitted project-generated artifacts:

- source artifact/parser integrity and `16/16` exact join;
- independent mechanism dossiers `0/30`;
- pre-outcome ambiguity sets `0/30`;
- independently reviewed discriminating actions `0/30`;
- hardware coverage `1/6` identities and one family;
- incident/channel coverage `16` incidents with two operational channels versus required `30`; and
- explicit raw-data/reproduction and rights limitations.

Counts use distinct incident and physical-identity keys. Windows, sensor channels, branches, simulator cases, or repeated records cannot inflate independent incidents, families, or hardware identities.

Passing parser, hash, signature, join, or sealed-truth separation checks proves only those checks. The conjunctive readiness rule remains failed and emits `BLOCKED_EVIDENCE`; it does not produce `null_result`, negative causal evidence, or permission to train.

### `inbar.physical-admission.0`

The prospective iteration-001 contract is not executable evidence because its acquisition authority is bootstrap-only, required control/validator identities contain placeholders, and no production receipt/pilot outcome exists. All-zero digests are invalid sentinels, not unresolved identities that can reach execution.

The physical admission rule requires, before any causal method:

- exact rights and raw-evidence custody;
- 30 complete physical root incidents under the same-incident completeness rule;
- three system families, six hardware identities, and three fault families;
- calibrated measurement/clock/frame/asset configuration records;
- independently reviewed mechanisms, ambiguity sets, discriminating safe actions, and outcome authority;
- separate safety, resource, execution, and physical interlock evidence; and
- frozen claim/stop/consequence rules.

Until then the only eligible wedge is offline/shadow dossier compilation and human-reviewable test recommendation. Recommendation is not experiment, execution, diagnosis, recovery, safety, or realized outcome.

### Rights as a hard scientific precondition

Unknown NASA ADAPT terms and the project’s raw-redistribution prohibition prevent raw-byte admission for the proposed use. The rights blocker is orthogonal to whether the data might be scientifically valuable. Missing raw bytes also narrow reproducibility. Odeya cannot cure either problem by retaining a digest, invoking fair-use language generically, or re-running consumed bootstrap authority.

## Cross-fixture invariant tests

The adapter suite must reject at least:

1. `planned_count` substituted for `observed_count`;
2. an absent episode imputed as zero or copied from its pair;
3. arm/pair/cluster mutation with unchanged reported result;
4. rounded display numbers used as verification oracle instead of exact result artifacts;
5. interval crossing zero compiled as equivalence, benefit, harm, or affirmative null;
6. mechanism activity compiled as outcome improvement;
7. committed receipt trusted over contradictory replay;
8. mutable corpus scan treated as immutable evidence;
9. partial correction failure used to erase independently valid correction facts;
10. diff locator/gold/post-hoc threshold omitted from exposure history;
11. missing judge outputs silently classified;
12. source `null` or `pass` label overriding Odeya validity;
13. channels/windows/simulator branches inflating physical incidents;
14. parser/signature/integrity pass granting scientific readiness;
15. bootstrap/placeholder identity granting execution;
16. recommendation compiled as performed physical action; and
17. rights-blocked bytes entering a model, artifact package, or publication.

Each mutation must fail at its intended validation layer. A parser failure is not evidence the statistical or physical gate detected the problem.

## Verification plan

For every admitted method:

1. freeze the exact source input manifest and rights decisions;
2. transcribe the source algorithm into a language-neutral registered rule with tests derived from, but not implemented by calling, the source analyzer;
3. implement two independent verifier paths where the consequence is claim-bearing;
4. generate synthetic/boundary/known-bad/metamorphic cases independently of the implementation under test;
5. compare exact typed result objects before presentation rounding;
6. run the real fixture in two clean network-denied environments with exact dependencies;
7. retain implementation/environment/resource/discrepancy records;
8. obtain statistical review for Sentinel/Telos semantics and physical/domain review for Inbar refusal/admission rules; and
9. freeze the method registry and fixture package digests in the architecture candidate.

Agreement does not prove assumptions. Reviewers must separately assess whether pairing, clustering, resampling, evidence-universe selection, exposure classification, physical completeness, and claim language are scientifically defensible.

## Acceptance boundary

The structural [`MethodRegistry`](../schemas/method-registry.schema.json) contract and a deliberately non-admitted Sentinel example now exist. This profile is sufficient for Gate A only after its placeholder identities are replaced by exact method objects, two genuinely independent implementations for the claim-bearing rules, executed vectors/results, source rights, environment identities, and expert determinations. Until then it is a precise adapter design, not method admission.

Even after acceptance, the first slice can establish only that Odeya represents and adjudicates these historical research distinctions correctly. It cannot establish autonomous research ability, physical-world competence, production isolation/recovery, general scientific superiority, or customer value.
