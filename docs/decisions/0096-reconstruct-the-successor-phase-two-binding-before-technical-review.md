## Context

ADR 0095 retained a source-separated, byte-bound successor chain at published
commit `86c0f1ed8ba20d74324d64529bf5435a0524f4cd`. Its validator passed fourteen
backing preimages, forty-eight chain known-bads, and 132 Python, Node.js, and
Java recomputations. A fresh-context refutation nevertheless found that the
retained phase-two authentication challenge ID belonged to the older ADR 0093
confirmation receipt.

The exact mismatch was:

- successor confirmation-receipt digest:
  `sha256:768e5446e6a5a8f3a3b510478739b20254843d1e9c6affec049a1faaa6a33458`;
- retained authentication challenge ID:
  `sha256:1b19ecdf21c6a4d9644fe18757cbd11d1bd32b4a0059a23b2e66ff7363e23a05`;
- independently reconstructed authentication challenge ID:
  `sha256:f02587cb898ab1e272f73295b1be167671a0382b98e7dd2f1e2a3344f5e0a6b2`.

The retained `1b19…` identity is correct only for predecessor receipt digest
`sha256:9ca484b4ec05698b8d41c3175b566b7e9bef28d41ee008b8561daeb90f2a35bc`.
The successor validator had compared the recorded committed receipt digest to
the actual receipt, but it copied the recorded authentication challenge ID
into its expected relation. It never reconstructed the phase-two binary frame
from the actual successor receipt bytes. Three evaluator implementations could
therefore agree while the cross-file challenge relation was stale.

This is a defect in synthetic architecture evidence and its validator. It is
not evidence that a real ceremony failed; no real ceremony occurred.

## Decision

### Reconstruct instead of copying

The generator and a separately implemented validator must reconstruct the
phase-two challenge from:

1. the exact retained v2 challenge-frame profile;
2. the retained shared inputs, phase intervals, and phase-two nonce;
3. the independently reconstructed presentation challenge ID;
4. the actual 368-byte successor confirmation-receipt frame; and
5. the exact confirmation-receipt profile ID.

The expected authentication frame is 943 bytes with raw digest
`sha256:0b1e54b802ada4262e05942a4325a4acc18107d2cb19cfaf8c50d706e466441c`.
Its challenge ID is `sha256:f02587cb898ab1e272f73295b1be167671a0382b98e7dd2f1e2a3344f5e0a6b2`.
Neither implementation may consume the recorded authentication challenge ID
as an input to this reconstruction.

### Permit one exact predecessor migration

The generator may migrate the stale vector only when both conditions hold:

- the complete vector file has predecessor digest
  `sha256:c81d7d9636d5a4f79a545bbad0ef5454ab8a4bb171d896e0035f1096b2c9cc1a`;
  and
- the complete relation equals the independently reconstructed relation except
  for the one exact stale authentication challenge ID.

Migration replaces that one same-length identifier while preserving every
other vector byte. An unrelated mutation, a second occurrence, a different
stale value, or any other relation disagreement is refused. Subsequent runs
require the corrected relation and reproduce the generated bytes exactly.

### Reissue changed downstream instances

Published candidate-instance identities are not repaired in place. The
unchanged schemas, ruleset, resolver profile, Evidence 0.2 fixture, backing
preimages, evaluator sources, and runtime profiles retain their identities.
Every changed downstream instance receives a new artifact identity:

- vector, normative-input, and input-manifest bindings use `.0002`;
- the backing-byte receipt uses `.0002`;
- all three projection and result identities use `.0002`;
- the comparison receipt uses `.0002`;
- the Seal fixture uses `.0003` while retaining the unissued Seal 0.2 schema;
  and
- the generation manifest uses `.0002`.

The generation manifest retains the exact published predecessor commit, tree,
sole parent, manifest binding, vector binding, stale value, and disposition.
The parent Git object remains the historical byte source; this correction does
not relabel it valid.

### Make the stale relation a named falsifier

The downstream corpus now contains forty-nine intent-bound known-bads. The new
case changes only the corrected authentication challenge ID back to the exact
predecessor value and requires these four diagnostics:

- `backing_confirmation_relation_authentication_challenge_id_mismatch`;
- `backing_confirmation_relation_mismatch`;
- `backing_reason_codes_mismatch`; and
- `backing_verification_disposition_mismatch`.

The first diagnostic is the intended guard. The remaining three prove that the
relation contradiction propagates into the receipt state rather than being
recorded as an inert warning.

### Bind technical review without a self-reference cycle

The corrected generation manifest defines
`hda-successor-t0-technical-review-scope.0002`. It binds the validator, vector
and answer corpora, chain known-bads, ruleset, resolver, challenge inputs,
source fixtures, toolchain lock, five schemas, generator, generated artifacts,
and their transitive source/configuration/lock members. The later review report
and status prose are excluded so the reviewed root does not contain its own
review digest.

The retained review report at
`architecture/human-decision-assurance-context-isolated-technical-review.json`
initially recorded two rounds and now preserves eleven append-only attack

1. the exact published predecessor was `refuted` by independent phase-two

2. rounds two through ten reissued each refuted boundary instead of hiding
   it; current root
   `sha256:97062b38a14d5bdccf5ad87c547c62388e7cd82256a445f631856aecee54e1d9`
   retains closure observation `.0003`, install observation `.0004`, and the
   corrected Gate and generator local-object plus strict-binding controls.
   Round ten refuted the live generator reader's repository-configuration

The bounded result is `no_grounded_refutation_observed_within_declared_attacks`.
It does not mean correct, approved,

The report is evidence from model workers, not a `ReviewDetermination`.
